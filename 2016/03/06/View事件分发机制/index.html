View事件分发机制

View事件流源码分析

  1. public boolean dispatchTouchEvent(MotionEvent ev)

        用来进行事件的分发，如果事件传递给当前的View，这个方法将在第一个被调用，返回结果受当前的onTouchEvent和下级View的dispatchTouchEvent方法的影响，表示是否消耗当前事件。

  2. public boolean onInterceptTouchEvent(MotionEvent event)

        在上述方法内部调用，用来判断是否拦截某个事件，如果当前View拦截了某个事件，那么在同一个事件序列当中，此方法不会被再次调用，返回结果表示是否拦截当前事件。

  3. public boolean onTouchEvent(MotionEvent event)

        在dispatchTouchEvent方法中调用，用来处理点击时间，返回结果表示是否消耗当前事件，如果不消耗，则在同一个事件序列中，当前View无法再次接收到事件。

    三种方法大致的关系如下面所示：

public boolean dispatchTouchEvent(MotionEvent ev) {
    boolean consume = false;
    //当前控件是否拦截事件
       if(onInterceptTouchEvent(ev)){
        /* 拦截事件，交给当前控件的onTouchEvent方法处理
         * 如果当前控件拦截事件之后并没有处理，则将事件向上传递，而不是传给子控件
         */
        consume = onTouchEvent(ev);
    } else {
        //不拦截事件，交给子控件处理
        consume = child.dispatchTouchEvent(ev);
    }
    return consume;
}

事件流分析

    当一个点击事件产生之后，它的传递过程如下：Activity->Window->View，即事件总是先传递给Activity，再由Activity传递给Window,再通过Window传递给顶级的View，也就是所谓的DecorView，再按照事件分发机制进行下去。如果一个View的onTouchEvent()方法返回false,那么它的父控件的onTouchEvent()将会被调用，最后Activity的onTouchEvent()会被调用。

关于事件传递tips

  1. 同一个事件序列是指手指接触屏幕到离开屏幕这之间发生的一系列事件。这个事件序列以down事件开始，中间有不定量的move事件，最后up事件结尾。
  2. 正常情况下，一个事件序列只能被一个View拦截并消耗。一个元素拦截了某个事件，那么同一个事件序列内的所有事件都会直接交给它处理，可通过特殊的手段来将一个事件序列给多个View处理。
  3. 某个View一旦决定拦截，那么之后的事件到来时不会去调用onInterceptTouchEvent，会直接交给这个View进行处理了。
  4. 某个View一旦开始处理事件，如果不消费ACTION_DOWN事件，同一事件序列都不会交给它来处理了，并且事件将给它的父控件进行处理，即父控件的onTouchEvent会被调用。
  5. ViewGroup默认不拦截任何事件，Android源码中ViewGroup的onInterceptTouchEvent方法默认返回false
  6. View没有onInterceptTouchEvent方法，事件传递过来的时候直接调用onTouchEvent方法。
  7. View的onTouchEvent方法默认都会消耗事件，除非它是不可点击的(clickable和longClickable同时为false)。
  8. onClick发生的前提是View是可点击的，并收到了down和up两个事件
  9. 事件的传递过程是由内向外的，即事件总是先给父元素，然后再由父元素分给子View，通过requestDisallowInterceptTouchEvent方法可以在子View中干预父元素的事件分发过程，但是ACTION_DOWN事件不受影响，代码的原因。

源码解析

Touch事件的产生

    Android系统的用户输入部分分为下面5个部分：

  1. Android的应用程序层：通过重新实现onTouchEvent和onTrachballEvent等函数来接受运动事件(MotionEvent)，通过重新实现onKeyDown和onKeyUp函数来接受按键事件(KeyEvent)。这些事件都包含在了android.view包中。
  2. Java框架层：在这一层中具有KeyInputDevice等类用于处理由EventHub穿上来的信息，通常信息由数据结构RawInputEvent和KeyEvent来表示。通常情况下，对于按钮事件由KeyEvent来穿送给应用程序层，对于触摸事件通过RawInputEvent传上来。在Java框架层的表现形式为InputEvent，是MotionEvent的父类。(EventHub是处于native框架层的一个libui库中的.cpp)
  3. KeyLayout和KeyCharacterMap文件：这两个文件与EventHub都属于native框架的libui，定义按键布局和按键字符映射需要运行时配置文件的支持，其配置文件的后缀名分别为：kl和kcm。
  4. EventHub：native框架中的libui的一部分，它实现了对驱动程序的控制，并从中获取信息。
  5. 驱动程序：/dex/input目录中，通常是Event类型的驱动程序。

    MotionEvent的实际存储位置在native内存中，一连串事件用一个单向链表来维护。具体怎么传上来的= =，还是去看老罗的博客吧。

事件的分发过程

  1. Activity对点击事件的分发过程
        Activity接受到事件之后，会传递给Window来完成事件的处理,Window再传给decorView来进行处理，decorView一般就是当前界面的底层ViewGroup。

    public boolean dispatchTouchEvent(MotionEvent ev) {
    if (ev.getAction() == MotionEvent.ACTION_DOWN) {
        onUserInteraction();
    }
    if (getWindow().superDispatchTouchEvent(ev)) {
        return true;
    }
    return onTouchEvent(ev);
}

        onUserInteraction()只是留出的一个接口，在Activity里面默认无操作。上面的代码可以看到Activity直接将操作传递给了Window来处理。如果Window那边整块没有操作，接着就会调用Activity的onTouchEvent()方法。

        接着进入Window类，里面只提供一个抽象方法public abstract boolean superDispatchTouchEvent(MotionEvent event);,找到唯一实现类PhoneWindow类，有如下代码：

    @Override
public boolean superDispatchTouchEvent(MotionEvent event) {
    return mDecor.superDispatchTouchEvent(event);
}

        Window又将事件传递给了手下的decorView去处理。DecorView继承自FrameLayout，是一个ViewGroup类，在Activity中执行setContentView()方法的时候实际上是向DecorView里面添加了一个子View，因此从这里开始事件向下传递。

  2. 顶级Viedw对点击事件的分发过程
        先进入ViewGroup类中查看dispatchTouchEvent()方法，这个方法是事件传入的起始点，dispatchTouchEvent()方法有点长，先看intercept部分：

        在ACTION_DOWN的时候或者mFirstTouchTarget != Null的时候，会去检查ViewGroup是否要拦截这个事件，里面会检查标志位FLAG_DISALLOW_INTERCEPT，这个标志位会标记其子View是否调用过requestDisallowInterceptTouchEvent方法，如果调用过则不会去拦截，如果没有调用，则会去调用onInterceptTouchEvent()方法来检查是否要拦截。当ViewGroup的子控件处理事件成功时，mFirstTouchTarget会被赋值并指向子元素。也就是说如果ViewGroup拦截了事件，intercepted的值为true,mFirstTouchTarget为null,这是一种情况，另外一种情况就是intercepted为false,mFirstTouchTarget不为null。下面分两种情况分开看不同的处理代码：

    • ViewGroup拦截了事件，intercepted的值为true,mFirstTouchTarget为null

          dispatchTransformedTouchEvent()方法里面如果传入的child为null,则会调用handled = super.dispatchTouchEvent(event);，里面会根据条件判断调用View的onTouchEvent()，也事件就是交给自己处理。

    • ViewGroup没有拦截了事件，intercepted为false,mFirstTouchTarget不为null

    • 如果这个方法dispatchTransformedTouchEvent返回true，则会在addTouchTarget方法中将child的值给mFirstTouchTarget，接下来的事件流会直接传给这个view进行处理

    • 如果返回了false，继续传递给其他的子View。
