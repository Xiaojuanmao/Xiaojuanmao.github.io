EventBus解析(一)简单介绍

题外话(可直接无视= =

    最近接触了一下EventBus这个开源项目，在好多人都已经用烂的情况下才开始。在了解之前也听到过好多种声音，有说eventbus是语法糖，越用越上瘾，不建议经常使用等等。也有说eventbus挺好用的，又省事来又过瘾，简直关爱代码从事人员的身心健康。是非多，两种声音并不能代表什么，只能说明这个项目大家的关注度还是不小的。

    github地址

    eventbus官方文档

简介

EventBus is publish/subscribe event bus optimized for Android

EventBus是Android平台上经过优化的、用来发布/订阅指定事件的事件传递机制

    盗用官网上的一张图，简单的介绍了eventbus是如何进行工作的。

优点

  • 简化组建之间的交流方式
    • 降低event的发送者和接受着之间的耦合程度
    • 能在Activity、Fragment以及后台的线程之间通信方面表现的很好
    • 避免了繁杂的依赖关系以及生命周期等问题
  • 让你的代码更加的简化
  • 速度快
  • 体积小
  • 好多app都集成在用 = =

简单使用

    怎么把EventBus弄到自己的项目里面来就不用说了，EventBus的使用很简单，只需要三步即可：

  1. 定义Event
        这里说的Event就是在Android编码的过程中，需要传递的JavaBean。

    public class MsgEvent{
    private String msg;
    public MsgEvent(String msg) {
        this.msg = msg;
    }
    public String getMsg() {
        return msg;
    }
}

  2. 注册EventBus以及Subscribers

    注册EventBus

        通过EventBus或者EventBusBuilder的一些方法能将一个类注册到EventBus实例中，EventBus将在注册过的类中寻找带有@Subscribe注解的方法。

    EventBus eventBus = EventBus.getDefault();
eventBus.register(this);

        在使用完之后需要注销掉

    eventBus.unregister(this);

        如果在Activity或者是Fragment中使用EventBus的话，可以和其声明周期进行绑定。

    注册Subscribers

        EventBus初始化完成之后，就可以定义类似于Receiver的事件处理者了,用注解@Subscribe来定义。

    @Subscribe
public void onEvent(MsgEvent event){
    /**
     */
}

  3. 发送event
        所有的准备工作都做好之后，就可以发送想要传递的数据，在上面注册的接受端处理就可以了。

    eventBus.post(new MsgEvent("hello world"));

四种工作模式

    上面的只是简单的使用，用的都是默认设置好的配置，EventBus还提供了四种ThreadMode来扩展功能，这四种功能使得EventBus不仅能够处理Event的传递，还能处理android上UI线程等方面的问题。可以通过@Subscribe注解来设定不同模式，下面介绍下这四种模式：

  1. POSTING

    @Subscribe(threadMode = ThreadMode.POSTING)
public void onMessage(MsgEvent event) {
/**
 */
}

        通过设定threadMode来指定ThreadMode为POSTING，这项参数表明订阅方法onMessage(MsgEvent event)的调用将在post线程完成，也就是说如果在UI thread线程调用eventBus.post()方法，则会在该线程调用onMessage(MsgEvent event)方法。

  2. MAIN

    @Subscribe(threadMode = ThreadMode.MAIN)
public void onMessage(MsgEvent event) {
/**
 */
}

        无论eventBus.post()方法在哪个线程调用，onMessage(MsgEvent event)方法都将在主线程被调用，如果本身就是UI thread调用了eventBus.post()，那就会被当作POSTING的情况处理了。

  3. BACKGROUND

    @Subscribe(threadMode = ThreadMode.BACKGROUND)
public void onMessage(MsgEvent event) {
/**
 */
}

        onMessage(MsgEvent event)方法将在后台的线程被偷偷的调用，如果调用eventBus.post()方法的线程不是UI thread，那么就直接偷懒的作为POSTING情况进行处理了，如果是UI thread的话，就会用另外一个线程来处理。

  4. ASYNC

    @Subscribe(threadMode = ThreadMode.ASYNC)
public void onMessage(MsgEvent event) {
/**
 */
}

        最后一个模式完全将event的发送和处理放在了两个不同的线程中。比较适用于一些耗时操作，例如网络请求、I/O等等。

Configuration

    前面用到的EventBus都是利用默认配置的，如果想自己改变配置可以利用内置的Builder来完成，例如下面的几种使用方法:

EventBus eventBus = EventBus.builder().logNoSubscriberMessages(false).sendNoSubscriberEvent(false).build();

EventBus eventBus = EventBus.builder().throwSubscriberException(true).build();

    还有其他的一些配置这里就不列出来了，在具体的使用过程中再去查阅。

Sticky Events

    还有个比较有意思的功能就是Sticky Events，官方这样解释的：

Instead of implementing your own caching, you can use sticky events. EventBus keeps the last sticky event of a certain type in memory.

用Sticky events可以替代自己实现缓存，意味着sticky events发出之后并不是马上得到处理就消失了，而是可以延时处理。

    Sticky Example

eventBus.postSticky(new MsgEvent("hello world"));

    Sticky event已经发出，新启动一个activity之后仍然可以处理这个事件，只要它没有被新的sticky event覆盖。例如：

@Override
public void onStart() {
    super.onStart();
    EventBus.getDefault().register(this);
}

@Subscribe(sticky = true, threadMode = ThreadMode.MAIN)
public void onEvent(MsgEvent event) {
    // UI updates must run on MainThread
    textField.setText(event.getMsg());
}

@Override
public void onStop() {
    EventBus.getDefault().unregister(this);
    super.onStop();
}

    只要在Sticky event被覆盖掉之前，都能够处理到该事件。关于sticky event被覆盖的情况，也就是每一个event类只能存在一个sticky。也可以随时的得到这个sticky event，用法如下：

MsgEvent stickyEvent = EventBus.getDefault().getStickyEvent(MsgEvent.class);
if (stickyEvent != null) {
    EventBus.getDefault().removeStickyEvent(stickyEvent);
}

Priorities and Event Cancellation

    Priorities

    当一个event注册有多个@Subscribe的时候，究竟先调用哪一个？EventBus提供了Priorities参数来表明他们之间先后调用的关系，默认的优先级为0，优先级越大越先调用，在之后的源码分析中看的会更加的明确，优先级的用法如下：

@Subscribe(priority = 1)
public void onEvent(MsgEvent event) {
...
}

    这个优先级的管辖范围只限于同一ThreadMode中，如果处于不同的ThreadMode中，优先级并不是那么完全的惯用，因为不同的模式中调用方式不同，并不能一定保证先后顺序。

    Cancelling event delivery

    发出一个event之后可能面临着需要取消的问题，eventbus也有留出接口cancelEventDelivery(Object event)，具体的用法如下：

@Subscribe
public void onEvent(MsgEvent event) {
    //...
    EventBus.getDefault().cancelEventDelivery(event);
}

    最好在高优先级的订阅方法中取消event的传递。

    EventBus上面的一些用法基本上就这么多，还有些功能可以到官网上去看看。下篇就开始从最顶端的API入手，分析EventBus的实现原理了。
